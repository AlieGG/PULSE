01 · Overview

What it is, in one breath

PULSE is a small Node.js service ("the Conductor") that joins the Pro DJ Link network as a virtual CDJ, listens to the beat grid coming off the Pioneer rig, and broadcasts a tiny set of synchronized variables to every Pixelblaze on the lighting network — so every LED segment in the room renders in perfect phase with the music, on every beat, on every bar, without ever asking the network for permission frame-by-frame.

On top of that runs a custom web-based control surface (PULSEDECK) — a tablet-friendly UI that replaces OSC/Pilot for human-driven scene selection, master brightness, hue offsets, kill switches, and zone control. It's the single pane of glass that ops uses live.

Two design-time tools complete the system: PULSEFORGE generates Pixelblaze patterns from natural-language prompts with live in-browser preview, geometry awareness, and contract validation. PULSEMAP is a visual canvas for laying out the physical rig — strips, matrices, vehicle-mounted runs — and exports both Pixelblaze pixel maps and the topology JSON that every other layer of PULSE consumes. Same data model across runtime and authoring.

v0.3 adds the creative depth and trust layers: the Prompt Library (mood wheel, genre presets, soul sliders, named animation idioms, A/B variant generation) turns PULSEFORGE from a code helper into a creative instrument. The Trust Stack (deterministic browser-based Pixelblaze simulator + AST/contract validator) is what makes AI-generated patterns shippable to stage hardware without manual review. And a LumenCanvas integration roadmap (CLASP protocol bus, LATCH visual flow programming) maps the path to standing on Moheeb's open infrastructure instead of hand-rolling forever.

Design philosophy

Centralize the beat detection. Decentralize the rendering. The Conductor publishes bpm + beatAnchorMs at low frequency. Each Pixelblaze computes its own phase using a Firestorm-synced clock. No per-frame network dependency. No drift between segments. Network can hiccup for seconds and the show keeps grooving on the last-known anchor.

Why this shape

The Conductor sits in the middle on purpose. BLT is the beat brain. Firestorm is the pattern catalog. The PBs are the renderers. PULSEDECK is the human. None of these speak each other's protocols natively. The Conductor is the only thing that needs to know all four, and it stays small (a few hundred lines of Node) so it's auditable, swappable, and easy to debug at 2am.

Beat phase timing diagram

This is the critical bit. Below shows how a single beat propagates and why each PB ends up phase-locked despite asynchronous arrival times.

sequenceDiagram
  participant DJM as Pioneer DJM
  participant BLT as Beat Link Trigger
  participant C as Conductor
  participant PB1 as Pixelblaze A
  participant PB2 as Pixelblaze B

  Note over DJM,PB2: Time T₀ — beat fires on Pioneer
  DJM->>BLT: Pro DJ Link beat packet
  BLT->>C: OSC /beat (bpm=128)
  Note over C: Captures now() as anchor
  par broadcast (parallel)
    C->>PB1: setVars {bpm, beatAnchorMs}
    C->>PB2: setVars {bpm, beatAnchorMs}
  end
  Note over PB1,PB2: Each PB now derives beatPhase LOCALLY from its Firestorm-synced clock
  loop every render frame
    PB1->>PB1: phase = ((time - anchor) mod interval) / interval
    PB2->>PB2: phase = ((time - anchor) mod interval) / interval
  end
  Note over PB1,PB2: Phase identical within ±5ms across all PBs
even if next setVars never arrives
The key insight

If we sent "now beat!" on every beat, jitter from WiFi, Node event loop, and PB packet processing would add a visible ±20-80ms wobble. By sending "the beat grid is at this anchor at this BPM" once per beat and letting each PB extrapolate from a synchronized clock, jitter disappears. Visual is rock solid even if a single message drops.

05 · The Conductor

What the bridge actually does

The Conductor is a single Node.js process. It runs forever, restarts on crash, holds four concerns: (1) listen to BLT over OSC, (2) maintain WebSocket connections to every Pixelblaze, (3) expose a WebSocket API for PULSEDECK, (4) talk to Firestorm's HTTP API when scenes need to swap.

Internal state machine

stateDiagram-v2
  [*] --> Booting
  Booting --> Discovering: PBs found via Firestorm
  Discovering --> LiveDJ: BLT sends /beat
  Discovering --> SensorMode: No BLT, sensor PB present
  Discovering --> FreeRun: No BLT, no sensor → manual BPM
  LiveDJ --> SensorMode: BLT silent > 5s
  SensorMode --> LiveDJ: BLT resumes
  SensorMode --> FreeRun: Sensor PB drops
  FreeRun --> LiveDJ: BLT resumes
  LiveDJ --> Panic: Operator hits kill
  SensorMode --> Panic: Operator hits kill
  FreeRun --> Panic: Operator hits kill
  Panic --> LiveDJ: Operator clears

Three fallback modes

  • Live DJ mode — BLT is feeding beats. This is the normal state. BPM, beat anchor, downbeat, bar, track signature are all flowing.
  • Sensor mode — BLT silent for 5+ seconds (DJ unplugged the CDJ, network issue, between sets). The Conductor reads frequencyData + energyAverage from a designated "leader" Pixelblaze with the Sensor Board and synthesizes a coarse BPM via autocorrelation, or simply forwards energy as energy setVar and stops sending beat anchors. Patterns gracefully degrade.
  • Free run mode — No DJ, no sensor. PULSEDECK exposes a "tap tempo" pad. Operator taps and that becomes the BPM. Useful for soundcheck, walk-in music played from the venue PA, ambient sets.

The core code sketch

const osc = require('osc')
const WebSocket = require('ws')
const { discoverPixelblazes } = require('./firestorm-client')

let bpm = 128, beatAnchorMs = Date.now(), bar = 0, mode = 'freerun'
let pbs = []  // WebSocket connections, keyed by ID

// 1. discover Pixelblazes via Firestorm
discoverPixelblazes().then(list => {
  pbs = list.map(pb => ({ id: pb.id, ip: pb.ip, ws: connect(pb.ip) }))
})

// 2. OSC inbound from BLT
const udp = new osc.UDPPort({ localPort: 9000 })
udp.on('message', msg => {
  if (msg.address === '/beat') {
    bpm = msg.args[0]
    beatAnchorMs = Date.now()
    mode = 'live'
    broadcastVars({ bpm, beatAnchorMs, mode: 1 })
  }
  if (msg.address === '/bar') bar = msg.args[0]
  if (msg.address === '/track/changed') handleTrackChange(msg.args[0])
})
udp.open()

// 3. WebSocket API for PULSEDECK
const wss = new WebSocket.Server({ port: 8080 })
wss.on('connection', ws => {
  ws.on('message', raw => {
    const cmd = JSON.parse(raw)
    if (cmd.type === 'scene') setScene(cmd.scene)
    if (cmd.type === 'brightness') setBrightness(cmd.zones, cmd.value)
    if (cmd.type === 'kill') panicAllBlack()
    if (cmd.type === 'tap') handleTapTempo(cmd.t)
  })
})

// 4. heartbeat — keeps PBs aware we're alive even between beats
setInterval(() => broadcastVars({ bpm, beatAnchorMs, mode: modeNum(mode) }), 250)

// 5. watchdog — switch to fallback if BLT silent
setInterval(() => {
  if (Date.now() - lastBltMs > 5000 && mode === 'live') mode = 'sensor'
}, 1000)
Why a 250ms heartbeat

Pixelblaze WebSocket connections will drop without traffic. A cheap 250ms heartbeat with the current bpm+anchor keeps every socket alive, makes reconnection unnecessary in most cases, and lets PBs detect Conductor death (no traffic for 2s → switch to last-known-good autonomous mode).

UI feature list

Top status bar. Always visible. Tells you in one glance: BLT connected? Firestorm up? How many PBs online vs configured? Current mode? Live BPM with a pulsing orb that visually beats. If any pill goes red, ops sees it instantly without leaving the active scene.

Now Playing card. Pulls track name + player number from BLT track metadata. Shows the auto-scene assigned to this track (if any). Shows current bar position. This is huge for ops — they know what the DJ is doing without watching the CDJs.

Scene grid. 8 quick-launch slots (Q1–Q8), expandable to banks via swipe. Tap to launch. Long-press to edit (rename, pick PB program from Firestorm catalog, set default colors, set per-zone overrides). Active scene is outlined in cyan. Color swatches give a visual cue at a glance. This is the bulk of live ops.

Zone strip. One tile per logical zone (could be "Stage L", "Truss", "Panthers Vehicle Front", whatever you've defined). Shows online/offline status, current brightness as a bar. Tap a tile to drill in: per-zone scene override, per-zone brightness, per-zone hue offset. Critical for events where you want different looks in different rooms.

Side controls. Master brightness (sliders, not buttons — for smooth fades). Hue offset (rotates the active palette in real time). Strobe rate (gates the beatPhase variable, lets you do 1/2, 1/4, 1/8 beat strobes without changing scene). Beat divider locks the visual rhythm to a multiple of the source beat — useful when DJ is at 174 dnb and you want the lights at half time.

Mode lock. Forces the Conductor into Live / Sensor / Tap mode, overriding the auto-fallback. Useful for soundcheck (force Tap), or when you know DJ is taking a break (force Sensor so it reacts to background music).

Kill switch. Big red button. Tap → instant black across the entire rig. Tap again → restore previous state. This must work even if everything else is broken. Implementation note: this sends a special kill=1 setVar that every pattern is required to honor in its render function. Belt and suspenders: also sends Firestorm a "brightness 0 to all" command.

Tech stack for PULSEDECK

  • Frontend: Single-page app, vanilla JS or lightweight framework (Preact, Svelte). No build step ideally — serve raw HTML/CSS/JS from the Conductor's static directory. Easier to debug, faster to load on a venue tablet.
  • State: WebSocket to the Conductor for everything. UI reflects Conductor state, not local state — so two tablets stay in sync, and a reload doesn't lose context.
  • Persistence: Scenes, zone assignments, color palettes saved as JSON on the Conductor's disk. Versioned. Git-trackable.
  • PWA: Add manifest + service worker so PULSEDECK can be "installed" full-screen on iPad. Hides Safari chrome, looks native.
  • Auth: Local network only by default. Optional shared password if exposing on venue WiFi.
Why not just use OSC/Pilot

OSC/Pilot is great but it (a) costs money, (b) requires its own configuration export per show, (c) speaks OSC only — every state change is one-way, no feedback into the UI, (d) doesn't know which scene is active or whether PBs are online. PULSEDECK is purpose-built: it shows the rig's state because it is the rig's UI, not a control surface bolted onto it.

07 · Pattern Contract

What every PULSE pattern must implement

For a Pixelblaze pattern to participate in the system, it needs to declare a small set of exported variables that the Conductor will write to, and respect a few conventions in its render loop. This is the contract — keep it stable across patterns and the whole catalog stays composable.

Required exported variables

// ────── PULSE protocol v1 ──────
export var bpm = 128          // set by Conductor
export var beatAnchorMs = 0    // ms timestamp of last beat
export var bar = 0             // 1..N within phrase
export var downbeat = 0        // 1 for one frame on bar 1
export var energy = 0          // 0..1 audio energy (sensor mode)
export var mode = 1            // 1=live, 2=sensor, 3=freerun, 0=panic

// ────── PULSEDECK overrides ──────
export var masterBrightness = 1 // 0..1
export var hueOffset = 0       // 0..1, rotates palette
export var strobeRate = 1      // 1=no strobe, 2,4,8 = subdivide beat
export var beatDivider = 1     // half-time / double-time visuals
export var killActive = 0      // 1 = force all black

Local beat-phase computation

var beatPhase, barPhase, nowMs

export function beforeRender(delta) {
  // shared Firestorm-synced clock in ms
  nowMs = time(0.015259) * 65536

  var interval = 60000 / (bpm * beatDivider)
  beatPhase = ((nowMs - beatAnchorMs) % interval) / interval

  // strobe gate: only "on" during the strobeRate-th of each beat
  if (strobeRate > 1) {
    beatPhase = (beatPhase * strobeRate) % 1
  }
}

export function render(index) {
  if (killActive) { rgb(0,0,0); return }

  // ... pattern-specific code using beatPhase, energy, etc ...
  var h = (myHue + hueOffset) % 1
  var v = brightness * masterBrightness
  hsv(h, 1, v)
}
Contract discipline

Every pattern must honor killActive in its first render line, and must multiply final brightness by masterBrightness. These are non-negotiable. A pattern that ignores them is a pattern that bricks the kill switch — which is the worst possible failure mode. Add a linter / test script that grep-checks every pattern in the catalog for both.

08 · Reliability

What makes this rock solid

The whole rig has to survive the worst night of its life — flaky venue WiFi, a DJ that yanks the ethernet to swap CDJs, a Conductor that crashes mid-set, a tablet that goes to sleep, a PB that loses power. Below is what each layer is designed to absorb.

Operational hardening

  • systemd wraps the Conductor with Restart=always, RestartSec=2, journal logging. Crashes are invisible to ops.
  • Watchdog timer — if the Conductor process is alive but not emitting heartbeats (event loop blocked), an external Python script SIGKILLs and restarts it.
  • Health endpoint at /healthz on the Conductor returns JSON of all state. PULSEDECK polls this every 2s; if it goes red, operator sees it.
  • Structured logging — every state transition (mode change, scene swap, PB connect/disconnect) is logged as JSON with timestamps. Post-show debugging becomes trivial.
  • Configuration in version control — scenes, zone definitions, PB IP list all live as JSON in a git repo. Every show is a tagged commit. Rollback is a checkout.
  • Static IPs for PBs via DHCP reservation on the AP. No mDNS race conditions. Conductor knows exactly which IP is which zone.
  • Network segregation — DJ Pro DJ Link, PB WiFi, and venue WiFi are on three separate L2 networks. The Conductor is the only thing that bridges them. No accidental cross-traffic.
  • UPS on Conductor + switch + AP for ~15 min runtime. Venue power blips don't kill the show.
  • Soundcheck script — a CLI that walks every PB, verifies WebSocket connectivity, sends a test pattern, confirms expected pixelCount. Runs in <30s before doors open.

Catalog discipline

The rig is only as reliable as the patterns it runs. Adopt these conventions:

  • Every pattern starts from the same boilerplate (see §07).
  • Every pattern includes a header comment block: name, author, date, intended zone type (1D strip, 2D matrix, etc.), expected pixelCount range.
  • Patterns are versioned in a git repo. The Firestorm catalog is generated from the repo, not edited live on PBs.
  • A "linter" script verifies killActive handling and masterBrightness multiplication before any pattern is committed.
  • Each pattern has a "rest mode" — if energy < 0.05 and no beats for 30s, the pattern should ease into a calm idle state rather than going dark or strobing. Polished.

Do not

11 · PULSEFORGE

AI-driven pattern composition

The catalog problem: a stage rig that's only as interesting as the patterns it can play. Hand-writing Pixelblaze patterns is fun but slow, and the per-show iteration cost is real — different venues, different fixture geometries, different musical moods all want new looks. PULSEFORGE is the in-browser tool that lets you describe a pattern in plain English, generates a PULSE-compliant Pixelblaze program with full geometry awareness, validates it against the contract, previews it live in the browser, and one-click deploys it to a target zone or saves it to the catalog.

What you get

A web app served by the Conductor at /forge. Pick a target zone from the PULSEMAP topology, write a prompt like "slow purple wave that punches harder on the downbeat, drifts toward magenta on the third bar of every phrase", and seconds later you've got working code, a live animated preview, and a "deploy to Zone B" button. Iterate by chatting: "slower" · "more contrast" · "add a sparkle on energy spikes". Save the good ones to the catalog where PULSEDECK can launch them.

// Slow Purple Punch — generated 2026-05-15
// Target: Zone B (1D strip, 144px)
export var bpm = 128
export var beatAnchorMs = 0
export var bar = 0
export var masterBrightness = 1
export var hueOffset = 0

export function beforeRender(d) {

  var v = w * (0.4 + punch * 0.6) * masterBrightness
  hsv(h, 1, v)
}

The system prompt

Claude needs three things every time: a tight description of the Pixelblaze language subset (it's not regular JavaScript — 16.16 fixed point, no closures, no objects, no let/const, no switch), the PULSE protocol contract (required exported variables, kill switch handling, master brightness multiplication), and the target zone's geometry. Geometry awareness is the secret sauce — a pattern for a 5×7 matrix uses render2D(i,x,y), a pattern for a vehicle wraparound strip wants the actual aspect ratio. Without geometry, you get generic 1D ramps.

You are generating code for the Pixelblaze LED controller running
the PULSE protocol. The language is a JS-like subset with these rules:

  - All numbers are 16.16 fixed-point. No integers vs floats.
  - No closures, no objects, no `let` or `const`, no `switch`.
  - `export var` declares pattern parameters and external inputs.
  - `beforeRender(delta)` runs once per frame; `render(i)` per pixel.
  - For 2D maps: `render2D(i, x, y)` where x,y are 0..1.
  - For 3D: `render3D(i, x, y, z)`.

EVERY pattern MUST:
  1. Declare: bpm, beatAnchorMs, bar, masterBrightness,
     hueOffset, killActive (see contract).
  2. Check killActive first in render() and return rgb(0,0,0).
  3. Multiply final brightness by masterBrightness.
  4. Compute beatPhase locally from time() and beatAnchorMs.

Target zone:
  - id: zone_b_truss
  - geometry: 1D strip
  - pixelCount: 144
  - physical length: 3.0m

  [insert 3 catalog patterns matched by tag similarity]

Output ONLY the Pixelblaze code. No prose, no markdown fences.

The validator (the trust layer)

Generated code never goes directly to a Pixelblaze. The validator sits between the model and the device and enforces the contract — it's a small parser + linter written once and reused for every generation. It catches the things that would silently break the show.

  • AST parse against the Pixelblaze grammar — bad syntax fails fast without hitting the device.
  • Exported var check — confirms all required PULSE protocol vars are declared.
  • Kill switch check — grep-asserts that render() reads killActive in its first statement.
  • Master brightness check — asserts masterBrightness is multiplied into the final color call.
  • Geometry sanity — if zone is 2D, pattern must use render2D; rejected otherwise.
  • Code size — Pixelblaze has memory limits; reject anything over the device limit.
  • Auto-retry — if validation fails, the validation errors are fed back to Claude with a "fix these specific issues" prompt. Usually clears in 1-2 round trips.

The browser simulator

Before code touches a real PB, it runs in the browser. A small JavaScript-shim implementation of the Pixelblaze runtime — exported vars become writable, time(), wave(), hsv(), rgb(), render(), beforeRender() all implemented in real JS. The simulator advances time, fires fake beat anchors at the chosen BPM, calls render(i) for each pixel, and pushes the resulting colors to a canvas. For 1D zones, a horizontal strip. For 2D, a grid. For 3D, a Three.js scene with a draggable camera.

Why simulate, not deploy-and-look

Deploying every iteration to real hardware is slow, requires the rig powered on, and gets distracting for anyone in the room. Simulator is instant, runs in the same browser tab as the prompt input, and is good enough to dial in a pattern before going to silicon. Final visual confirmation still happens on the real fixture before catalog commit, but 80% of the iteration loop never touches WiFi.

Catalog and deployment

Save the good patterns. Catalog entries are JSON files in a git repo, one per pattern, with metadata: name, author, description, target zone type, BPM range, energy range, tags, the prompt that produced it, the generated code. PULSEDECK reads the catalog to populate its scene grid. Firestorm reads the catalog to dispatch programName changes. Every commit is a versioned snapshot of the show's pattern library.

Iteration UX

The refinement chat is conversational. Each user turn is appended to the prompt history along with the previous pattern code; Claude returns a delta or rewrite. Quick-chip buttons (slower, more saturation, sparkle on energy) are macros that pre-fill the refinement input — they're not hardcoded edits, they're prompt sugar. A pattern goes through 3-8 iterations on average before it's catalog-ready, and the simulator means each iteration is <5 seconds.

A note on AI quality

Claude is genuinely good at Pixelblaze patterns — they're small, math-flavored, and well-documented. The catch is consistency across the contract. Without the validator + system prompt discipline, you'll get patterns that "work" but quietly skip the kill switch or forget hue offset. With them, the generated code is production-grade and ships to the rig without manual review. The discipline is what makes it trustworthy live.

12 · PULSEMAP

Physical LED mapping

Every other piece of PULSE depends on knowing where the pixels actually are. Pixelblaze needs pixel maps to render 2D and 3D patterns correctly. PULSEDECK needs topology to show a live rig view. PULSEFORGE needs geometry to generate patterns that fit the fixture. PULSEMAP is the visual editor that produces all of it from a single source of truth — and it does what hand-written Pixelblaze map JS can't: handle complex installations like a multi-channel animatronic vehicle in a single pane of glass.

Why it exists

Pixelblaze ships with a mapper that's basically a JavaScript text editor. For a single strip it's fine; for a 5×7 matrix it's tolerable; for a multi-zone build with strips wrapping curved surfaces across multiple output expander channels, it's a punishing way to spend a Tuesday. PULSEMAP replaces the text editor with a canvas: drop in a reference photo, trace LED paths visually, assign controller/channel/start index per segment, name the zone, click export. Out the other side: a pixel-map.js per Pixelblaze (uploaded automatically via Firestorm's API), and a topology.json consumed by the Conductor.

Calibration mode

The single most useful feature for large physical builds. Click Calibrate, click any segment, and PULSEMAP walks the segment's pixels live on the fixture — pixel 0 lights red, then pixel 1, then pixel 2, advancing every 500ms — so you can visually verify that the digital map matches the physical wiring. If pixel 0 in software lights up where you expected pixel 299, you know the segment direction needs to be reversed. Calibration uses the existing Conductor → PB WebSocket so no extra plumbing.

A more elaborate mode "binary search" lights half the segment at a time, asks the operator which half is which?, and rebuilds direction/order from physical observation. Saves hours on a vehicle build with hard-to-trace runs.

Importing reference images

Drop a photo into the canvas. It becomes a ghost layer at adjustable opacity. Set scale by clicking two points and entering the real-world distance between them. Now your tracings have real units. Critical for vehicle builds where you're often working from a CAD drawing or a photo of the welded chassis. Multiple images can stack (front view + side view) for 3D builds — the 3D mode lets you place those as orthographic backgrounds on the X/Y/Z planes.

Integration with the rest of PULSE

  • PULSEDECK rig view — a tab in PULSEDECK shows the PULSEMAP canvas read-only, with live status overlaid. Offline segments go grey. The currently-active scene colors the segments in their broad palette so you can see at a glance what's playing where. Pure observability win.
  • PULSEFORGE geometry awareness — generation prompts auto-include the selected zone's geometry (1D length, 2D rows/cols, 3D bounds). Patterns generated against Zone B are guaranteed to use 1D rendering; patterns generated against the eyes matrix use render2D with the right serpentine handling.
  • Conductor zone routing — when PULSEDECK says "set Zone Z_BODY to Slow Purple Punch," the Conductor looks up which controllers/channels belong to Z_BODY and dispatches to those specific PBs. Without the topology, scenes would be all-or-nothing.
  • Soundcheck script — the <30s pre-show check walks every segment from the topology, validates pixel counts match what the PB reports, flags mismatches before doors open.
Source of truth

topology.json is committed to the same git repo as the pattern catalog. Every change to the physical rig is a tagged commit. If the Panthers vehicle gets a new strip welded on next month, you re-map, commit, and every other layer of PULSE auto-picks up the change at next deploy. No "did we update the map in three places" failures.

13 · Prompt Library

Composing what you want to feel

PULSEFORGE without scaffolding works fine — type a sentence, get a pattern. With scaffolding it becomes a creative instrument. The model still does the generation — the library gives intent a vocabulary that matches the way a designer thinks.

The Mood Wheel

A 2D pad based on Russell's circumplex of affect — the cleanest map between emotional language and dimensional parameters. The vertical axis is activation (calm ↔ energetic); the horizontal is valence (cool ↔ warm). Drag the dot anywhere on the wheel and PULSEFORGE auto-appends the corresponding mood descriptors to the prompt. A dot in the top-right corner adds "euphoric, driving, hot, urgent"; bottom-left adds "reflective, deep, quiet, drifting". The wheel removes the need to remember the vocabulary — it gives the designer a directional control instead of a blank text box.

Genre Presets

Each genre is a curated bundle of defaults: a BPM range (DnB locks 165-180; House 118-128), a palette family (Trance loves cyan/purple, Techno wants white/orange/red, Disco wants saturated rainbow), and a motion vocabulary (Ambient → drifting, breathing, slow; Techno → mechanical, geometric, punctuated; Hip-Hop → bouncing, syncopated, weighty). Selecting a genre primes the model with three things at once instead of asking the designer to spell each out. The soul sliders can then override individual dimensions.

Soul Sliders

Six abstract dials that map to multiple low-level parameters at once. Energy drives speed, brightness peaks, frequency of motion; Complexity adds noise layers, parallax, harmonic motion; Warmth shifts the palette toward red/orange/yellow; Surprise introduces stochastic events (sparkles, flickers, color jumps); Contrast widens the dynamic range between dim and bright; Smoothness shapes the easing curves (sharp triangle vs sine vs cubic eased). These are the controls a designer instinctively wants — the prompt library translates them into model directives in the system prompt.

Animation Idiom Library

Named patterns with established emotional resonance: slow breathing for ambient warmth, downbeat punch for kick emphasis, rain sparkle for high-frequency twinkle on energy peaks, fire flicker for organic randomness, wave pulse for harmonic motion across the strip, spectrum bloom for color-explosion-on-drop, mirror sweep for symmetrical horizontal motion. Each idiom is a tested code fragment that the model uses as a structural anchor — pick an idiom and the generation is guaranteed to feel like that idiom. New idioms get added whenever a generated pattern works particularly well; the library grows with the rig's life. This is also the seam where community-contributed patterns plug in — share idioms, share looks.

A/B Variant Generation

Every Compose click generates four variants in parallel. Each variant subtly varies one or two dimensions while keeping the rest constant — A might be the pure interpretation, B with more contrast, C with a different palette anchor, D with slower easing. Click any variant to make it the canonical pattern, or chain another Compose with a refinement to evolve it further. Cuts decision fatigue, surfaces happy accidents, and feels closer to choosing a take in a recording session than coding.

Reference Library

Every committed pattern lives in the catalog with metadata — mood wheel position, soul-slider state, genre, idiom anchors, prompt, generation timestamp. Browse the catalog by emotional quadrant: "show me everything in the energetic-cool quadrant" for the next drop. Click any pattern to load its full creative context back into PULSEFORGE — the mood, the sliders, the genre, the idioms — and use it as a starting point. The catalog becomes the rig's personality, captured and reusable. Search by tag, by author, by date, by "patterns that worked at the last show."

Music-aware composition (v0.4)

Drop a track into PULSEFORGE; the browser analyzes it with Web Audio + Meyda (frequency bins, RMS energy, transient detection) and identifies sections: intro / verse / build / drop / breakdown / outro. Each section gets its own auto-composed pattern variation that shares a palette but varies energy, sparked by the track's actual energy curve. The result is a pre-choreographed lighting score that travels with the track. Powerful for fixed sets (festival headliners with rehearsed mixes) and installations (museums, art cars, brand events) where the music is curated.

Why this works for emotion-evoking output

Models are limited by the words you give them. "Make it pretty" gives you mediocre pretty. "Slow breathing, magenta-violet bloom, 30% warmth, 60% surprise, on a 120-BPM build" gives you something specific enough to feel intended. The prompt library is structured intent — it stops the operator from being a prompt engineer and lets them be a designer with a richer palette of inputs. The model gets cleaner instructions; the human gets a more expressive instrument.

14a · The Validator

A small deterministic Node module. Takes Pixelblaze source + zone geometry, returns a structured report of pass/fail with specific issues. No magic. Same module runs in PULSEFORGE during generation, as a pre-commit git hook on the pattern catalog repo, and as a CLI sweep (pulse-lint catalog/) before every show.

Implementation

const { parse } = require('./pb-grammar')
const rules = require('./rules')

function validate(source, zoneGeometry) {
  var issues = []
  var ast
  try { ast = parse(source) }
  catch (e) {
    return [{ rule: 'P001', severity: 'error', msg: e.message, line: e.line }]
  }

  for (const rule of rules) {
    const hits = rule.check(ast, source, zoneGeometry)
    if (hits) issues.push(...hits)
  }
  return issues
}

// Structured issues become a fix prompt for auto-retry
function issuesToFixPrompt(issues) {
  return "The pattern has these problems. Fix them all:\n" +
    issues.map(i => "- [" + i.rule + "] " + i.msg).join("\n")
}

module.exports = { validate, issuesToFixPrompt }

14b · The Simulator

A faithful reimplementation of the Pixelblaze runtime in TypeScript, running entirely in the browser. The Validator says "this code is correct"; the Simulator says "this code looks like this." Different jobs, both required.

What it implements

  • Pattern lifecycle — globals, export var, beforeRender(delta), render(i) / render2D(i,x,y) / render3D(i,x,y,z).
  • Built-in functions — full set: wave, square, triangle, sin, cos, tan, sqrt, abs, min, max, floor, ceil, round, pow, random, noise (Perlin), time(interval), hsv, rgb, hsl2rgb, resetTransform, translate, rotate, scale.
  • Fixed-point semantics — values clamped to 16.16 range; division and overflow behave like the device, so visual bugs in simulation mirror device bugs.
  • Time + beat injection — Simulator writes bpm and beatAnchorMs as if a Conductor were feeding them; operator can scrub BPM in the UI to test how a pattern behaves at 100 / 140 / 174.
  • Geometry mounts — 1D → horizontal strip of N colored dots; 2D → Canvas2D grid; 3D → Three.js scene with the topology's actual pixel positions and a draggable camera.

Architecture — transpile, don't interpret

Naive approach: tree-walking interpreter. Slow — won't hit 60fps for a 300-pixel strip. Better: parse PB source to AST, emit equivalent safe JavaScript (PB built-ins become shim function calls), new Function() it once, then call the resulting render loop natively. This gets 200-500fps for typical patterns on a laptop. Transpile happens once per generation; the runtime is then near-native JavaScript.

Code skeleton

import { transpile } from './transpile'
import { mkRuntime } from './shim'

export class Simulator {
  private render!: (i: number) => [number, number, number]
  private beforeRender!: (dt: number) => void
  private rt = mkRuntime()                // time(), wave(), hsv()...
  private lastT = performance.now()

  load(source: string, geom: Geometry, pixelCount: number) {
    const js = transpile(source, geom)
    const mod = new Function('rt', js + '; return {render, beforeRender}')
    const exports = mod(this.rt)
    this.render = exports.render
    this.beforeRender = exports.beforeRender
    this.rt.pixelCount = pixelCount
  }

  setBeat(bpm: number, anchorMs: number) {
    this.rt.bpm = bpm
    this.rt.beatAnchorMs = anchorMs
  }

  step(buffer: Uint8ClampedArray) {
    const now = performance.now()
    const dt = now - this.lastT
    this.lastT = now
    this.beforeRender(dt)
    for (let i = 0; i < this.rt.pixelCount; i++) {
      const [r, g, b] = this.render(i)
      buffer[i*4+0] = r*255
      buffer[i*4+1] = g*255
      buffer[i*4+2] = b*255
      buffer[i*4+3] = 255
    }
  }
}

Performance budget

  • Target: 60fps stable for a 300-pixel 1D pattern on a 2024 mid-range laptop.
  • Acceptable: 30fps for a 35×35 = 1225-pixel 2D pattern. UI shows a "preview is slowing for performance" badge.
  • 3D Three.js render is rate-limited to 30fps regardless; visual feedback is still adequate.
  • Simulator runs in a Web Worker so PULSEFORGE's chat UI stays responsive during render.

Why both

The Validator without the Simulator gives you syntactically valid patterns that look like nothing intended. The Simulator without the Validator means you'll occasionally generate code that crashes the Pixelblaze. Together, they make the AI loop trustworthy: generate → gate → visualize → approve → deploy. The operator never sees broken code, never sees code they haven't already watched run. The trust stack is what turns "AI-generated patterns" from a parlor trick into a stage-grade tool.

15 · LumenCanvas Integration

Standing on Moheeb's shoulders

PULSE as drafted has a custom Conductor that hand-rolls OSC ingestion, Pixelblaze WebSocket plumbing, clock distribution, and state sync. It works. But Moheeb Zara's LumenCanvas projects — CLASP (the protocol) and LATCH (the visual flow programming environment) — solve a large slice of that as battle-tested open infrastructure. Integration isn't required for v1, but the architecture should leave room for it because every PULSE feature gets cheaper and more interoperable once the bus is CLASP-shaped.

What CLASP brings

CLASP (Creative Low-latency Application Streaming Protocol) is a Rust-based router with TypeScript / Python / Arduino client SDKs, MIT/Apache licensed. Five things matter for PULSE:

  • Semantic signal types. CLASP distinguishes Params (stateful, persisted, delivered to late-joiners), Events (confirmed, ephemeral), Streams (high-rate, droppable), Gestures (phased begin/update/end), and Timelines (keyframed automation). PULSE's bpm/beatAnchorMs are Params. Beat ticks are Events. Energy/spectrum data is a Stream. The kill switch is a Param. The typing already exists and the routing layer respects it.
  • Protocol bridges as architecture. Built-in bidirectional bridges for OSC, MIDI, Art-Net, DMX, MQTT, WebSocket, Socket.IO, BLE, Serial, HTTP. BLT → OSC → CLASP becomes one config line, not a custom Node listener. DMX out for non-PB fixtures is one more config line. The Conductor stops needing to know about ten protocols.
  • Late-joiner state sync. When a tablet reloads PULSEDECK mid-show, CLASP delivers the current state of every Param the moment it subscribes. No "what scene is active right now?" round-trip. Eliminates an entire class of "I refreshed and lost context" bugs.
  • Clock sync as a primitive. CLASP targets ±1ms LAN / ±5-10ms WiFi clock synchronization across clients. That's the same target PULSE's time()-via-Firestorm trick achieves, but it's a property of the bus rather than a bespoke pattern.
  • E2E encryption + delegatable auth. Future-proofing. When a touring rig needs to talk to a home-base monitoring dashboard across the public internet, AES-256-GCM and Ed25519 capability tokens are already there.

The Conductor's job collapses. Instead of OSC listener + custom WebSocket fanout + Firestorm HTTP shim + heartbeat + state sync, it becomes a single-purpose Pixelblaze adapter: subscribes to /pulse/** on the CLASP router, forwards setVars to the PBs over their native WebSocket. That's it. Everything else — OSC ingestion, MIDI, DMX out, multi-client state, late-joiner sync, encryption — is the bus's job. PULSEDECK, PULSEFORGE, LATCH, and the Conductor are all just @clasp-to/sdk clients addressing the same paths.

LATCH as alternate authoring

PULSEFORGE is prompt-driven. LATCH is node-driven — TouchDesigner / Max / Pd in the browser, with first-class CLASP nodes (Connect, Subscribe, Set, Get, Emit, Stream, Bundle) and an Electron desktop build. For designers who think visually and want to wire audio analysis → math → color → CLASP set rather than describe in words, LATCH plugs in as another authoring path. A LATCH flow that reads the venue's audio input, extracts bass energy via Meyda, threshold-triggers a sparkle, and publishes to /pulse/scene/launch is six nodes. Runs in the same browser tab as PULSEDECK if you want; or as an Electron app on the FOH machine. The two authoring tools are complementary — prompts for narrative-driven scenes, node graphs for generative/reactive flows.

conduyt — the firmware play

conduyt is Moheeb's host-to-device binary protocol — one firmware library handles framing, dispatch, and capabilities across multiple host SDKs. Pixelblaze firmware is closed-source so we can't drop conduyt in directly, but for any custom controllers built into the Panthers vehicle or future installations (ESP32-based fixtures, custom sensor nodes, animatronic actuators), conduyt gives you a real protocol instead of bespoke serial framing every time. v3 territory — only relevant once PULSE is hosting non-PB hardware.

A note on building on top of friends' work

CLASP and LATCH are both MIT/Apache licensed — open infrastructure, no commercial entanglement. Building PULSE to be CLASP-compatible from v1.5 onward isn't a dependency you take on; it's interoperability you gain. Two creative-technology stacks built by people who know each other, talking the same protocol, runnable independently or together. That's the right shape for the Miami / Mesa maker-tech axis.

What could still bite

RiskLikelihoodMitigation
Pioneer firmware update breaks BLTLow — 10yr track recordHold off updates until BLT confirms; CDJs work fine without updates.
WiFi AP saturation at >20 PBsMediumUse 5 GHz dedicated AP; chunk PBs across SSIDs if needed; or use PB ethernet expander board.

Open questions for v0.4

  • Pattern catalog scope — start with 12 patterns or 30? What's the minimum viable set for one festival weekend?
  • Should PULSEDECK have a recording mode that captures every scene/brightness/kill action with timestamps, so post-show you can replay or analyze what looked good when?
  • Resolume/MagicQ users get pre-cued scenes that fade in. Do we want a "next scene" / "go" workflow on top of the quick-launch grid?
  • Per-track scene presets — keyed by BLT track signature. Worth building, or premature optimization?
  • DMX bridge — does the Panthers vehicle build need DMX out for non-PB fixtures? If so, configure the CLASP DMX bridge in v1.5 or hand-roll Art-Net in v1.
  • PULSEFORGE model choice — Sonnet 4 is the starting point. Opus for complex 3D patterns? Local Gemma/Qwen for offline venues with no internet?
  • Music-aware composition — drop a track into PULSEFORGE, get a pre-choreographed lighting score by section. Useful for fixed sets, installations. v0.4 candidate.
  • PULSEMAP collaboration — multiple people editing the same topology at once during a build day? Likely v1 is single-editor with git as merge tool; CRDT-based realtime co-editing is a v2 polish item (and falls out for free once on CLASP).
  • Calibration robotics — for very large builds, a phone camera + AR could auto-detect physical pixel positions by watching the calibration sequence. Wild but doable; v3 territory.
  • Community idiom library — should the Animation Idiom Library be a shared public catalog where the Pixelblaze community contributes? Could be hosted at pulse.hello-miami.community or similar.
  • Catalog as a federated CLASP namespace — once on CLASP v2, the pattern catalog can be a federated Param tree. Other rigs running PULSE could pull-share looks. Risky for show consistency, powerful for community.
